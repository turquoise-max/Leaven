# 📋 직원 온보딩 및 관리 기능 개발 명세

## 1. 개요
본 문서는 직원의 매장 합류 과정(Onboarding)을 개선하고, 점주가 직원을 효율적으로 관리할 수 있도록 수기 등록 및 승인 프로세스를 구현하기 위한 개발 명세서입니다.

**핵심 개념 (Data Structure Strategy):**
> *"Discord의 '계정 프로필'과 '서버 프로필'의 관계와 유사합니다."*
> 
*   **계정 프로필 (`profiles`):** 앱 전체에서 사용되는 유저의 고유 정보 (내 정보).
*   **매장 프로필 (`store_members`):** 특정 매장 내에서 관리되는 직원 정보 (직원 명부).
*   **동기화 전략:** 온보딩 시 입력된 정보는 **계정 프로필을 우선 채우고**, 동시에 **매장 프로필에도 스냅샷으로 저장**됩니다. 이후 두 정보는 독립적으로 관리될 수 있습니다.

---

## 2. 주요 기능 및 구현 상세

### A. 직원 온보딩 프로세스 개선 (Scenario C)
**목표:** 매장 코드로 가입하려는 직원의 신원을 점주가 식별할 수 있도록 함.

1.  **화면:** `/onboarding/staff`
2.  **변경 사항:**
    *   기존: 매장 코드(6자리)만 입력.
    *   **변경:** 매장 코드 + **이름(필수)** + **전화번호(선택)** 입력 폼 제공.
3.  **로직 (`joinStoreByCode`):**
    *   **Step 1. 계정 프로필 업데이트:** 입력받은 `name`, `phone`이 있고 기존 `profiles` 정보가 비어있다면 업데이트.
    *   **Step 2. 매장 프로필 생성:** `store_members` 테이블에 `name`, `phone`을 포함하여 레코드 생성.
    *   상태: `pending_approval` (승인 대기).

### B. 점주용 직원 추가 기능 (수기 등록)
**목표:** 계정이 없는 직원도 점주가 직접 시스템에 등록하여 관리할 수 있도록 함.

1.  **UI:** `InviteStaffDialog` (직원 추가 팝업)
2.  **구성:** `Tabs` 컴포넌트를 사용하여 두 가지 방식 제공.
    *   **Tab 1: [초대하기]** (기존)
        *   이메일 입력 -> 초대장 발송.
    *   **Tab 2: [직접 등록]** (신규)
        *   **이름(필수)**, 이메일(선택), 전화번호(선택), 역할(기본: Staff), 시급 정보 입력.
        *   등록 시 `store_members`에 `user_id` 없이 레코드 생성.
        *   상태: `active` (즉시 근무 가능 상태로 간주).

### C. 가입 요청 승인/거절 프로세스
**목표:** 점주가 대기 중인 가입 요청을 인지하고 처리할 수 있도록 함.

1.  **알림 위젯 (Dashboard Main):**
    *   위치: 대시보드 메인 화면 상단.
    *   내용: "🔔 승인 대기 중인 요청이 **N건** 있습니다." (클릭 시 직원 관리 페이지로 이동).
2.  **승인 대기 목록 (Staff List):**
    *   위치: `StaffManagementPage` 상단 섹션.
    *   내용: 요청한 직원의 이름, 전화번호, 신청일 표시.
    *   Action:
        *   **[승인]:** `status`를 `active`로 변경. (해당 직원은 즉시 대시보드 접근 가능)
        *   **[거절]:** 해당 `store_members` 레코드 삭제. (직원은 다시 초기 화면으로 리셋됨)

### D. 직원 목록 표시 개선
**목표:** 수기 등록된 직원(프로필 없음)과 일반 직원(프로필 있음)을 모두 정상적으로 표시.

1.  **로직 (`StaffList`):**
    *   이름 표시 우선순위: `profile.full_name` > `store_members.name`.
    *   연락처 표시 우선순위: `profile.phone` > `store_members.phone`.
    *   상태 배지: 수기 등록된 직원의 경우 "미연동" 또는 별도 표시 고려 (선택 사항).

## 3. 데이터베이스 상태 변화

| 구분 | 초기 상태 | Action | 최종 상태 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **매장 코드 가입** | `pending_approval` | 점주 승인 | `active` | `user_id` 있음 |
| **매장 코드 가입** | `pending_approval` | 점주 거절 | **Deleted** | 레코드 삭제됨 |
| **수기 등록** | `active` | 점주 등록 | `active` | `user_id` 없음 (NULL) |
| **이메일 초대** | `invited` | 직원 수락 | `active` | `user_id` 매핑됨 |
