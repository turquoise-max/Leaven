# ☕ Cafe-Bakery 솔루션: 회원가입 및 온보딩 로직 명세

## **1. Scenario A: 자발적 신규 가입 (잠재적 점주)**

> **상황:** 시스템에 정보가 전혀 없는 신규 유저가 스스로 가입하는 경우.
> 
- **Step 1. 회원가입/로그인:**
    - 소셜(Google/Kakao) 또는 이메일로 가입 완료.
- **Step 2. 역할 선택 (Role Selection):**
    - 온보딩 페이지 진입: "환영합니다! 어떤 목적으로 오셨나요?"
    - 질문: **"매장 소유주이신가요?"** → **[예]** 선택.
- **Step 3. 매장 등록 (Store Creation):**
    - 매장명, 사업자 번호, 주소 등 필수 정보 입력.
- **Step 4. 완료:**
    - 초기 세팅 완료 후 **Owner 권한**으로 매장 대시보드 진입.

---

## **2. Scenario B: 이메일 초대를 받은 직원**

> **상황:** 점주가 시스템에서 '이메일'을 입력하여 정식으로 초대한 경우.
> 
- **Step 1. 초대 수신:**
    - 이메일 수신: *"OOO 카페에서 당신을 초대했습니다. [수락하기]"*
- **Step 2. 회원가입/로그인:**
    - 링크 클릭 시 `invite_token`을 유지한 채 로그인 화면으로 진입.
    - *UX Tip:* 초대받은 이메일 주소와 동일한 소셜 계정(구글 등) 가입을 유도.
- **Step 3. 자동 매칭 (System Logic):**
    - 로그인/가입 완료 즉시 토큰 유효성 검증.
    - 해당 유저를 `StoreMember` 테이블의 **Active(활동 중)** 상태로 변경.
- **Step 4. 완료:**
    - 별도의 온보딩 절차(매장 찾기 등) 없이 **즉시 직원(Staff) 뷰 대시보드**로 이동.

---

## **3. Scenario C: 수기 등록된 직원 및 자발적 가입**

> **상황:** 
> 1. 점주가 이름만 적어두거나(수기 등록 - Ghost Profile), 
> 2. 직원이 초대를 기다리지 않고 스스로 매장 코드로 가입한 경우. **시스템은 이 유저가 누구인지 모르는 상태.**
> 

### **Phase 1. 직원(User)의 행동: "입장 및 노크"**

- **Step 1. 회원가입/로그인:**
    - 앱 설치 후 전화번호/카카오 등으로 가입.
- **Step 2. 역할 선택:**
    - 질문: **"매장 소유주이신가요?"** → **[아니오]** 선택.
- **Step 3. 매장 코드 입력:**
    - 화면: *"근무하시는 매장의 초대 코드를 입력해주세요."*
    - (하단 회색 텍스트): *"코드가 없으신가요? 점주님께 문의하거나 매장명을 검색해보세요."*
    - **입력 정보:** 매장 코드(예: `CAFE-8821`), **실명(필수)**, **전화번호(선택)**.
    - Action: 정보 입력 후 **[가입 요청]**.
- **Step 4. 대기 화면 (Feedback):**
    - 화면 메시지: *"**OOO 카페/베이커리**에 가입 요청을 보냈습니다. 점주님의 승인을 기다려주세요."*
    - 상태: 유저는 `Pending_Approval` 상태로 대기.

### **Phase 2. 점주(Owner)의 행동: "교통정리 (Mapping)"**

- **Step 1. 알림 수신:**
    - 점주 앱 푸시 알림 또는 대시보드 알림: *"새로운 직원(김철수)이 가입을 요청했습니다."*
- **Step 2. 승인/거절 UI 위치:**
    - **[대시보드] > [직원 관리]** 페이지 진입.
    - **상단 섹션:** **[가입 승인 대기 목록]**이 별도로 표시됨.
        - *승인 대기 중인 요청이 없을 경우 해당 섹션은 숨김 처리.*
- **Step 3. 의사 결정:**
    - 점주는 요청한 직원의 정보(가입 이름, 전화번호)를 보고 **[승인]** 또는 **[거절]** 버튼을 클릭.
    
    > **옵션 A: 승인 (신규/기존 병합)**
    > 
    > - Action: **[승인]** 버튼 클릭.
    > - (고도화 시): "기존 수기 등록된 직원과 연결하시겠습니까?" 팝업 -> 기존 직원 선택 시 데이터 병합.
    > - Result: `StoreMember` 테이블의 상태가 `Active`로 변경. 직원은 즉시 대시보드 접근 가능.
    
    > **옵션 B: 거절**
    > 
    > - Action: **[거절]** 버튼 클릭.
    > - Result: 해당 가입 요청 레코드 삭제. 직원은 다시 가입 요청을 보내야 함.

---

## **4. 데이터베이스 상태 변화 요약**

이 로직을 구현하기 위한 `StoreMember` 테이블의 상태(`status`) 변화 흐름

| 시나리오 | 초기 상태 | 중간 과정 | 최종 상태 |
| --- | --- | --- | --- |
| **A (점주)** | `None` | 매장 생성 시 자동 생성 | `Owner` |
| **B (이메일 초대)** | `Invited` (이메일 발송됨) | 토큰 검증 성공 | `Active` |
| **C (수기 등록 - 유령)** | `Ghost` (이름만 있음, User ID 없음) | - | **옵션 A 선택 시** `Active`로 병합 |
| **C (매장 코드로 가입)** | `None` | 코드 입력 시 `Pending` (승인 대기) | **점주 승인 시** `Active` |
