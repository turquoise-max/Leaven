# 🚀 Leaven 온보딩 및 홈 화면 개편 기획서 (v2.2)

## 1. 개요 (Overview)
현재의 '/home' 화면은 유저의 권한(점주/직원)이나 매장 보유 상태(단일 매장/다중 매장)를 고려하지 않은 공급자 중심의 UI(탭 뷰)로 제공되고 있습니다. 

본 기획은 **'초거대 B2B SaaS 수준의 매끄러운 사용자 경험(Seamless UX)'**을 제공하기 위해, **'지능형 홈 화면 라우팅'**과 **'매직 링크 및 전화번호 기반 자동 온보딩'**을 도입합니다. 특히 오작동이나 엣지 케이스(Edge Case) 상황에서도 유저가 길을 잃지 않도록 탄탄한 안전장치를 마련합니다.

---

## 2. 지능형 홈(Home) 화면 라우팅 (Smart Routing)
사용자가 로그인한 직후의 상황(Context)에 따라 화면을 완벽히 다르게(Adaptive) 분기합니다.

| 유저 컨텍스트 | 화면 구성 (Phase) | 동작 로직 (Routing & Action) |
| --- | --- | --- |
| **초기 진입자** <br/>(소속 매장 0개 & 초대장 0개) | **Phase 1. 역할 선택 뷰** | 복잡한 탭 없이 **[새 매장 개설(점주)]** 또는 **[초대 코드로 합류(직원)]** 대형 카드 2장만 노출. ➔ 유저가 스스로 목적을 선택하도록 직관적 유도. |
| **합류 대기 직원** <br/>(소속 매장 0개, 대기 1개) | **Phase 2. 대기 상태 뷰** | 점장님의 승인을 기다리거나, **"전자 근로계약서 서명 대기 중 ⏳"**이라는 상태 카드 단 1장만 노출. 점주에게 연락하거나 이메일을 확인하라는 안내 문구 표기. |
| **정착 유저 (대다수)** <br/>(소속 매장 딱 1개, 초대장 0개) | **Phase 3. 직행 버스 (Auto-Redirect)** | 홈 화면을 렌더링하지 않고(0.1초 만에 스킵), **해당 매장의 대시보드(`/dashboard`)로 즉시 강제 이동**. 출퇴근/스케줄 확인 동선 단축. |
| **다점포 점주 / 투잡** <br/>(소속 매장 2개 이상 등) | **Phase 4. 통합 허브 뷰** | 내가 소유/소속된 전체 **'매장 그리드 카드'** 노출. 화면 하단에 **[초대 코드로 합류]** 및 **[새 매장 만들기]** 버튼 제공. |

> 🛡️ **UX 예외 방어 (Bypass Flag)**: 대시보드 사이드바의 "다른 매장 보기(스위처)"를 누른 경우엔 `?bypass=true` 파라미터를 붙여 Phase 3(강제 리다이렉트)를 무시하고 Phase 4(통합 허브 뷰)를 띄워줍니다. 이로써 단일 매장 점주도 언제든 새 매장을 만들 수 있습니다.

## 3. 초격차 직원 초대 온보딩 시스템 (Seamless Onboarding)
기존의 복잡한 '6자리 코드 수동 입력'을 2선(Fallback)으로 미루고, 터치(또는 스캔) 한 번으로 가입이 완료되는 1선(Main) 시스템을 도입합니다.

### 🎯 3.1 메인 전략: 매직 초대 링크 & QR 스캔
1. **점주 액션**: 매장 관리 페이지에서 고유 URL(`leaven.app/join/[STORE_CODE]`)을 복사하여 카톡으로 전달하거나, **QR 코드**로 다운로드하여 매장 벽에 부착.
2. **직원 액션 (UX)**: 
   - 직원이 카톡 링크 클릭 또는 QR 스캔.
   - 앱/웹 접속 후 회원가입을 완료하면, 브라우저가 들고 있던 `STORE_CODE`를 활용해 **자동으로 합류 신청(Submit)** 처리. ➔ *코드 복붙 과정 완벽 생략.*

### 🎯 3.2 킬러 기능: 유령 프로필(Ghost Profile) 전화번호 자동 매칭
1. **점주 액션**: 점주가 직원 관리 페이지에서 **[직원 수기 등록]**으로 직원의 `이름`과 `전화번호`를 사전 등록. (DB상에 `user_id`가 null인 '유령 프로필' 생성)
2. **자동 매칭 (UX)**: 직원이 앱 가입 후 로그인 시, 시스템이 전화번호를 즉시 대조함.
   - 팝업: *"🎉 [A매장]에서 이미 귀하를 직원으로 등록했습니다! 합류하시겠습니까?"*
   - 직원이 [수락] 클릭 시 ➔ 기존 유령 프로필에 직원의 `user_id`가 연동(Merge)되며 **점주 승인 대기 없이 즉시 정식 직원(Active)으로 승격**.

## 4. 완벽한 엣지 케이스(Edge Case) 방어 전략

어떤 변수가 생겨도 시스템이 꼬이지 않도록 3중 안전장치를 마련합니다.

### 🚨 Case 1. 점주가 직원의 전화번호를 오타 낸 경우 (매칭 실패)
- **상황**: 점주가 수기 등록 시 010-1234-5678을 5679로 잘못 입력.
- **방어 로직**:
  - 직원이 가입해도 번호가 달라 '자동 매칭 팝업'이 뜨지 않음. 직원은 자연스럽게 **Phase 1(역할 선택 뷰)**에 머뭄.
  - 여기서 직원이 점장님이 준 **QR코드(매직 링크)**를 찍거나 **6자리 코드**를 수동 입력하여 가입 신청(Fallback 작동).
- **계정 병합(Merge) 플로우**: 
  - 직원이 코드로 합류 신청 시, 점주의 "대기 중" 목록에 등장함.
  - 점주가 해당 직원을 [승인]할 때 시스템 팝업 ➔ *"기존에 수기 등록한 '유령 직원(오타 난 프로필)'과 동일 인물인가요?"*
  - 점주가 **[네, 덮어씌우기(Merge)]** 선택 시 ➔ 기존 수기 등록 정보(시급, 스케줄 등)가 신규 직원 계정으로 완벽히 이관되며 오타 난 번호도 자동 갱신됨.

### 🚨 Case 2. 엉뚱한 타인이 번호 오타를 악용해 가입 탈취(Hijacking) 시도
- **상황**: 오타 난 전화번호(5679)를 실제로 사용하는 타인이 우연히 르븐에 가입하여 '자동 매칭 팝업'을 보게 됨.
- **방어 로직**:
  - '유령 프로필' 자동 매칭 시 전화번호만 보지 않고, 회원가입 시 입력한 **실명(이름)**이 수기 등록된 이름과 일치하는지 2차 교차 검증 수행.
  - 이름이 다를 경우 매칭 팝업 자체를 띄우지 않음으로써 악의적인 접근 원천 차단.

### 🚨 Case 3. 다중 매장에 동일한 전화번호로 동시 등록된 경우
- **상황**: A매장과 B매장 점주가 동시에 동일한 알바생을 수기 등록해 둔 상태.
- **방어 로직**:
  - 알바생이 가입 시 *"🎉 [A매장], [B매장] 2곳에서 초대 대기 중입니다"* 라며 다중 선택이 가능한 통합 합류 팝업을 띄움. 
  - 수락한 매장들만 각각의 유령 프로필과 병합(Merge)되어 즉시 Phase 4(허브 뷰)로 진입.

## 5. [핵심] 3차원 상태(Status) 관리 아키텍처 및 탭 노출 정책
현장 점장님들의 실제 업무 흐름(당장 낼모레 출근할 알바생 스케줄부터 짜야 하는 상황)을 반영하여, **"법적 온보딩(계약)"**과 **"시스템 온보딩(앱 가입)"**을 완전히 분리합니다.

### 🧩 5.1 상태(Status)의 3차원 분리
직원의 탭 소속은 오직 **`근무 상태(Active/Pending)`**로만 결정되며, 그 안에서 2개의 배지(Badge)로 계약 및 앱 연동 상태를 직관적으로 추적합니다.
- 🏢 **근무 상태 (Tab 분류 기준)**: 재직 중(`active`) / 합류 요청(`pending_approval`) / 퇴사(`inactive`)
- 📝 **계약 배지 (Legal Status)**: 🔴 미작성(`none`) / 🟡 서명 대기(`sent`) / 🟢 체결 완료(`signed`)
- 📱 **앱 연동 배지 (App Status)**: ⚪ 미연동 수기등록(`user_id: null`) / 🟢 앱 연동됨(`user_id` 존재)

### 📌 Tab 1. 재직자 (Active Roster)
**"우리 매장에서 당장 스케줄을 뛰고 시급을 받는 모든 찐 직원 명부"**입니다.
1. **수기 등록(유령 계정) 직원**: 
   - 점장님이 "내일 출근해!" 하고 이름과 시급만 등록한 직원.
   - 앱 가입이나 계약서 서명을 안 했어도 **생성 즉시 무조건 재직자 탭**에 올라갑니다.
   - 뱃지 상태: `[⚪ 앱 미연동]`, `[🔴 계약서 미작성]` (점장은 이 직원을 스케줄표에 바로 넣을 수 있음)
2. **합류가 승인된 정식 직원**:
   - 점장님이 합류 요청을 수락(`[승인]`)한 순간 즉시 재직자 탭으로 넘어옵니다.
   - 뱃지 상태: `[🟢 앱 연동됨]`, `[🔴 계약서 미작성]` (점장님은 시간 날 때 프로필을 눌러 전자계약서를 쏘면 됨)

### 📌 Tab 2. 합류 요청함 (Requests Inbox)
기존의 '대기열' 개념을 **"현관문(Inbox)"**으로 축소합니다.
- 매직 링크나 6자리 코드를 치고 매장에 막 들어오려고 노크(`pending_approval`)하는 사람들만 모여 있습니다.
- 점장님이 **[수락]**을 누르는 즉시 `Tab 1(재직자)`로 튕겨 나갑니다. (계약서 작성은 이곳에서 강제되지 않습니다.)
- 만약 점장님이 이메일로 초대한 사람(`invited`)이 있다면, 그 응답을 기다리는 용도로만 쓰입니다.

### 📌 Tab 3. 퇴사자 (Resigned)
과거에 일했지만 현재는 퇴사 처리된 직원 (`status: inactive` & `resigned_at` 존재). 퇴사 처리 시 계약 정보와 과거 스케줄 기록은 영구 보존됩니다.
